What Are Supabase Edge Functions?
Server-side TypeScript functions distributed globally, running on the Deno runtime.

Definition

Supabase Edge Functions are server-side TypeScript functions that run on Deno's V8 isolates, deployed globally at the edge -- close to your users. They handle HTTP requests, webhooks, and scheduled events without you managing any infrastructure.

Think of them as your app's backend logic layer: authentication, payment processing, external API calls, image processing, and anything that shouldn't run on-device.

How They Differ from AWS Lambda / Cloud Functions

AspectSupabase Edge FunctionsAWS Lambda / GCF
RuntimeDeno (V8 isolates)Node.js, Python, Go, Java, etc.
DeploymentEdge (multi-region, close to users)Regional (single region default)
LanguageTypeScript/JavaScript firstMulti-language
Cold StartsMinimal (ESZip format)Variable (seconds)
Supabase IntegrationBuilt-in (Auth, DB, Storage)Manual SDK setup
InfrastructureZero config, fully managedMore configuration needed

The Deno Runtime

Deno is a modern JS/TS runtime built on V8, created by Ryan Dahl (who also created Node.js). Supabase chose Deno because:

  • Security by default -- sandboxed with no file/network access unless granted
  • TypeScript first-class -- no build step needed, native TS support
  • Web-standard APIs -- uses fetch, Request, Response (portable code)
  • ESZip bundling -- compact bundles for fast cold starts (97% improvement)
  • npm compatible -- import npm packages via npm: specifier

Key Features at a Glance

  • Auto-injected Supabase credentials -- SUPABASE_URL, SUPABASE_ANON_KEY, SUPABASE_SERVICE_ROLE_KEY are always available
  • Secrets management -- supabase secrets set KEY=VALUE for API keys
  • CORS helpers -- SDK provides importable CORS headers
  • Cron scheduling -- via pg_cron + pg_net
  • Webhook receivers -- Stripe, GitHub, Twilio etc.
  • npm packages -- import Stripe from 'npm:stripe@13'

Architecture & How They Work
From HTTP request to response -- the edge deployment model and Supabase service integration.

Request Flow

How a request travels through the Supabase Edge
Mobile App
(KMP Client)
→
Edge Gateway
(JWT + Routing)
→
V8 Isolate
(Deno Runtime)
→
Supabase Services
(DB/Auth/Storage)
  1. Client sends HTTP request to https://<project-ref>.supabase.co/functions/v1/<function-name>
  2. The edge gateway handles routing and JWT validation
  3. Request is forwarded to an available V8 isolate
  4. Your TypeScript function executes and returns a Response

Cold Starts vs Warm Execution

AspectDetail
Cold startNew isolate boots from ESZip bundle -- typically under 200ms
Warm executionIsolate stays active for subsequent requests (no boot overhead)
ScalingMultiple isolates run simultaneously per edge location
CleanupAt 50% resource usage, isolate finishes current request then shuts down

Limits & Constraints

LimitValueNote
CPU time/request2 secondsActual computation (excludes I/O wait)
Wall clock time400 secondsTotal elapsed including async I/O
Memory150 MBPer isolate
Bundle size20 MBAfter ESZip bundling
FilesystemEphemeralNo persistent storage across invocations
⚠️
The 2-second CPU limit means Edge Functions are not for heavy computation. Image ML inference, video processing, or bulk data crunching should use dedicated compute. But most STEP app needs (API calls, DB queries, auth) fit comfortably.

When to Use vs. Alternatives

ApproachBest For
Edge FunctionsCustom HTTP endpoints, external API calls, webhooks, complex TypeScript logic
Database Functions (RPC)Data transformation close to the data, triggers
RLS PoliciesRow-level access control and authorization
Database WebhooksReacting to INSERT/UPDATE/DELETE events

In practice, these are complementary. You'll use RLS for authorization, database functions for data-centric logic, and Edge Functions for everything requiring external communication.

10 Use Cases for Your STEP App
Each mapped to your current codebase -- Characters, Posts, Camera, GPS, and Favorites tabs. Click any card to see implementation details.
1

User Authentication

CRITICAL

Foundation for everything. No auth exists today -- all data is anonymous and device-local.

Current state: No user identity. Favorites and photos are device-bound.

Edge Function role: Secure server-side token validation, OAuth token exchange, session refresh. An auth-handler function processes sign-up, sign-in, and refresh actions.

KMP integration: Add SupabaseAuthRepository in commonMain. Store tokens via expect/actual TokenStorage (EncryptedSharedPreferences on Android, Keychain on iOS).

Unlock: Every subsequent use case depends on this.

2

Photo Upload & Processing

HIGH

Upload camera photos to Supabase Storage, generate thumbnails, strip EXIF GPS data.

Current state: CapturedPhotoEntity saves photos to local filesystem with GPS. Device-bound, no backup.

Edge Function role: photo-upload function receives base64 image, strips EXIF for privacy, generates thumbnail, uploads both to Storage, stores metadata in DB.

KMP integration: Add PhotoSyncUseCase that reads local file, base64-encodes, POSTs to edge function. Add synced column to SQLDelight table.

3

Cloud Sync for Favorites

HIGH

Sync SQLDelight favorites to Supabase across devices with conflict resolution.

Current state: FavoriteRepositoryImpl stores favorites in local Favorite table keyed by character_id. No cross-device sync.

Edge Function role: sync-favorites implements last-write-wins merge. Client sends local favorites + last sync timestamp, server merges and returns authoritative state.

KMP integration: Add FavoriteSyncService that runs on app launch and after each toggle. Add updated_at, deleted columns to Favorite table.

4

Posts API Migration

HIGH

Replace the fake GraphQLZero API with real persistent Supabase-backed posts.

Current state: PostRepositoryImpl does CRUD via GraphQLZero -- a mock API that never persists. Posts vanish between sessions.

Edge Function role: posts-api handles GET/POST/PUT/DELETE with real Postgres persistence, user ownership, and pagination.

KMP integration: Replace PostRepositoryImpl with SupabasePostRepositoryImpl using Ktor. The PostRepository interface stays identical -- zero UI changes needed.

5

Enhanced Character Search

MEDIUM

Full-text search with typo tolerance using Postgres pg_trgm + tsvector.

Current state: Rick and Morty API's name filter is simple substring match. "rik" returns nothing.

Edge Function role: character-search queries a cached character table with fuzzy matching. Searching "rik" or "morthy" returns ranked results.

6

Push Notifications

MEDIUM

Server-side FCM/APNs dispatch triggered by database webhooks.

Current state: No push notifications. No re-engagement capability.

Edge Function role: send-notification securely holds FCM/APNs credentials. Triggered by DB webhooks (e.g., new post inserted).

7

Analytics & Event Logging

MEDIUM

Lightweight server-side event collection -- no third-party SDK needed.

Current state: Zero analytics. No visibility into feature usage.

Edge Function role: track-event receives batched events, sanitizes PII, inserts into analytics_events table.

KMP integration: Add AnalyticsService singleton that buffers events and flushes every 30s or 20 events.

8

Rate Limiting / API Gateway

MEDIUM

Protect external APIs, cache responses, hide API keys.

Current state: Both Apollo clients call external APIs directly with no rate limiting or caching.

Edge Function role: api-gateway centralizes rate limiting (30 req/min), caches responses (5min TTL), keeps API keys server-side.

9

Scheduled Data Refresh

LOW

Cron job refreshes the character catalog into Supabase every 6 hours.

Current state: Characters fetched live on every app open. If the API goes down, Characters tab is broken.

Edge Function role: refresh-characters called via pg_cron every 6 hours, pulls all pages from Rick and Morty API, upserts into Supabase Postgres.

10

Social Features

LOW

Comments, likes, and sharing on posts with server-enforced rules.

Current state: No user interaction. Posts are solo content.

Edge Function role: social function handles like toggle (atomic, duplicate-safe), comment submission (sanitized, length-limited), and comment retrieval.

Implementation Roadmap
Recommended build order based on dependencies and value. Auth is the foundation -- everything builds from there.
1

User Authentication

Foundation. Add auth-handler edge function + SupabaseAuthRepository in KMP. Enables user identity for all subsequent features. Unlocks everything.

2

Posts API Migration

Replace fake GraphQLZero with real Supabase-backed posts-api. Easiest migration -- PostRepository interface is already clean. Real persistence = real value.

3

Favorites Cloud Sync

Apply the sync pattern from posts to favorites. sync-favorites with last-write-wins merge. Users keep favorites across devices.

4

Analytics & Event Logging

Quick win -- works even without auth (anonymous events). Gives you data to prioritize what to build next.

5

Rate Limiting / API Gateway

Protect external APIs, add response caching. Change Apollo's serverUrl to point at your edge function -- no other code changes.

6

Scheduled Character Refresh

Set up pg_cron to call refresh-characters every 6 hours. Decouples app from Rick and Morty API availability.

7

Enhanced Character Search

Build on the cached character table from step 6. Add pg_trgm index for fuzzy search. "rik" now finds "Rick".

8

Photo Upload & Processing

Higher complexity -- base64 encoding, Storage buckets, thumbnail generation. Build after the simpler features are stable.

9

Push Notifications

Requires FCM/APNs setup, device token registration, and database webhooks. High value but high setup cost.

10

Social Features

Comments, likes, sharing. Depends on auth + posts migration being stable. Transforms STEP into a community platform.

Setup & Your First Edge Function
From installing the CLI to running your first function locally -- a hands-on walkthrough.

Step 1: Install the Supabase CLI

bash
# npm (recommended for CI/CD)
npm install -g supabase

# Or Homebrew
brew install supabase/tap/supabase

# Verify
supabase --version

Step 2: Initialize & Link

bash
# Inside your KMP_Basic_App directory
supabase init           # Creates supabase/ directory
supabase login          # Authenticate (opens browser)
supabase link --project-ref <your-project-ref>
🔎
Find your project ref in the Supabase Dashboard under Project Settings > General. It's the alphanumeric string in your project URL.

Step 3: Create Your First Function

bash
supabase functions new hello-world

This creates supabase/functions/hello-world/index.ts:

supabase/functions/hello-world/index.ts
Deno.serve(async (req: Request) => {
  // Parse the JSON body
  const { name } = await req.json()

  // Your business logic
  const data = {
    message: `Hello ${name}!`,
    timestamp: new Date().toISOString(),
  }

  // Return JSON response
  return new Response(JSON.stringify(data), {
    headers: { 'Content-Type': 'application/json' },
  })
})

Step 4: Run Locally

bash
supabase start           # Start local Supabase stack
supabase functions serve  # Serve edge functions (hot reload!)

Step 5: Test It

bash
curl -i -X POST http://localhost:54321/functions/v1/hello-world \
  -H "Authorization: Bearer <anon-key-from-supabase-status>" \
  -H "Content-Type: application/json" \
  -d '{"name": "STEP App"}'
🎉
You just ran a serverless function locally! Changes to index.ts are picked up instantly -- save the file and curl again.

Step 6: Deploy to Production

bash
supabase functions deploy hello-world
# Now live at:
# https://<project-ref>.supabase.co/functions/v1/hello-world

Working with Supabase Services
Database, Auth, and Storage -- all accessible from within your Edge Functions via auto-injected credentials.

Two Client Patterns

Respects Row Level Security. Use when acting on behalf of an authenticated user.

typescript
const supabase = createClient(
  Deno.env.get('SUPABASE_URL')!,
  Deno.env.get('SUPABASE_ANON_KEY')!,
  {
    global: {
      headers: { Authorization: req.headers.get('Authorization')! },
    },
  }
)

Bypasses RLS entirely. Use for server-side operations needing full DB access.

typescript
const supabaseAdmin = createClient(
  Deno.env.get('SUPABASE_URL')!,
  Deno.env.get('SUPABASE_SERVICE_ROLE_KEY')!
)
🛑
Never expose SUPABASE_SERVICE_ROLE_KEY to the client. It has full admin access.

Database: Read & Write

supabase/functions/posts-api/index.ts
// INSERT a new post
const { data, error } = await supabase
  .from('posts')
  .insert({ title, body, user_id: userId })
  .select()
  .single()

// SELECT with pagination
const { data, count } = await supabase
  .from('posts')
  .select('*', { count: 'exact' })
  .order('created_at', { ascending: false })
  .range(0, 9)

CORS (Required for Browser Calls)

supabase/functions/_shared/cors.ts
export const corsHeaders = {
  'Access-Control-Allow-Origin': '*',
  'Access-Control-Allow-Headers':
    'authorization, x-client-info, apikey, content-type',
  'Access-Control-Allow-Methods': 'POST, GET, OPTIONS, PUT, DELETE',
}
Usage in every function
import { corsHeaders } from '../_shared/cors.ts'

Deno.serve(async (req) => {
  // MUST be first!
  if (req.method === 'OPTIONS') {
    return new Response('ok', { headers: corsHeaders })
  }
  // ... your logic ...
  return new Response(JSON.stringify(data), {
    headers: { ...corsHeaders, 'Content-Type': 'application/json' },
  })
})

Cron Scheduling (pg_cron)

sql
-- Schedule character refresh every 6 hours

  'refresh-characters',
  '0 */6 * * *',

  SELECT net.http_post(
    url    := 'https://<ref>.supabase.co/functions/v1/refresh-characters',
    headers := jsonb_build_object(
      'Authorization', 'Bearer ' || '<SERVICE_ROLE_KEY>'
    )
  );
  $$

Database Webhooks

Trigger an Edge Function when data changes:

sql
-- Fire notification on new post
CREATE TRIGGER on_post_created
  AFTER INSERT ON posts
  FOR EACH ROW
  EXECUTE FUNCTION supabase_functions.http_request(

    'POST', '{"Content-Type":"application/json"}',
    '{}', '5000'
  );

Step 3: Call Edge Functions

commonMain/.../data/repository/SupabasePostRepositoryImpl.kt
@Serializable
data class CreatePostRequest(val title: String, val body: String)

class SupabasePostRepositoryImpl(
    private val supabase: SupabaseClient
) : PostRepository {

    override suspend fun createPost(title: String, body: String): Result<Post> =
        runCatching {
            val response = supabase.functions.invoke(
                function = "posts-api",
                body = CreatePostRequest(title, body)
            )
            response.body<PostDto>().toDomain()
        }
}
💡
When Auth is installed on the same client as Functions, the SDK automatically injects the session's access token as an Authorization: Bearer header. No manual header management needed!

Step 4: Auth Flow

commonMain/.../data/repository/AuthRepositoryImpl.kt
class AuthRepositoryImpl(private val supabase: SupabaseClient) : AuthRepository {

    override val isAuthenticated: Flow<Boolean>
        get() = supabase.auth.sessionStatus.map {
            it is SessionStatus.Authenticated
        }

    override suspend fun signIn(email: String, password: String) =
        runCatching {
            supabase.auth.signInWith(Email) {
                this.email = email
                this.password = password
            }
        }
}

Migration Strategy & Next Steps
A 4-phase plan to migrate from Apollo GraphQL to Supabase Edge Functions, plus your deployment checklist.

4-Phase Migration Plan

1

Phase 1: Add Supabase Alongside GraphQL (Week 1-2)

Add supabase-kt dependencies without removing Apollo. Both clients coexist in Koin. New features use Supabase; existing features stay on GraphQL.

2

Phase 2: Feature Flag Abstraction (Week 2-3)

Introduce FeatureFlags + RepositoryFactory pattern to route requests per feature. This gives you instant rollback capability.

RepositoryFactory pattern
class PostRepositoryFactory(
    private val apolloRepo: ApolloPostRepositoryImpl,
    private val supabaseRepo: SupabasePostRepositoryImpl
) : PostRepository {

    private val active: PostRepository
        get() = if (FeatureFlags.useSupabasePosts) supabaseRepo else apolloRepo

    override suspend fun getPosts() = active.getPosts()
    override suspend fun createPost(...) = active.createPost(...)
}
3

Phase 3: Migrate Feature by Feature (Week 3-6)

For each feature: write Edge Function (TypeScript) -> implement SupabaseXxxRepositoryImpl (Kotlin) -> enable flag in staging -> validate -> flip in prod.

4

Phase 4: Remove Apollo (Week 7+)

Once stable: remove FeatureFlags/Factory wrappers, delete Apollo dependencies from libs.versions.toml and build.gradle.kts, delete .graphql schema files.

🔑
Zero UI changes required. Your clean architecture (repository interfaces in domain, Koin DI) means swapping Apollo for Supabase only touches the data layer. Use cases and screens remain untouched.
